# ステアリング原則

steeringファイルは**プロジェクトメモリ**であり、網羅的な仕様書ではない。

## ゴールデンルール

> 「新しいコードが既存パターンに従うなら、steeringの更新は不要」

## コンテンツ粒度

### 記録すべきもの

| 対象 | 例 |
|------|-----|
| 組織パターン | feature-first, layered |
| 命名規約 | PascalCaseルール |
| インポート戦略 | 絶対パス vs 相対パス |
| アーキテクチャ決定 | 状態管理方針 |
| 技術標準 | 主要フレームワーク |

### 記録してはいけないもの

| 対象 | 理由 |
|------|------|
| 完全なファイルリスト | カタログ化は保守負荷が高い |
| 全コンポーネントの説明 | パターンで代替可能 |
| 全依存関係 | package.json等で管理される |
| 実装詳細 | コードが真実 |
| エージェント固有ディレクトリ | `.cursor/`, `.gemini/`, `.claude/` 等 |
| `.kiro/settings/` の内容 | メタデータであり知識ではない |

## セキュリティ

以下を絶対に含めない：
- APIキー、パスワード、クレデンシャル
- データベースURL、内部IP
- シークレットや機密データ

## 品質基準

| 基準 | 説明 |
|------|------|
| 単一ドメイン | 1ファイル1トピック |
| 具体的な例 | コードでパターンを示す |
| 理由の説明 | なぜその決定をしたか |
| 保守可能なサイズ | 100-200行を目安 |

## 更新ルール（Syncモード）

- ユーザーセクションとカスタム例を保持する
- デフォルトは追加的更新（add, don't replace）
- `updated_at` タイムスタンプを付与する
- 変更理由を記録する
